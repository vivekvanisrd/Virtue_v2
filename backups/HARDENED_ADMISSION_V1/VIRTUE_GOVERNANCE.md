# PaVa-EDUX Master Architecture & Branding Standard (2026-27)

This document defines the core identity, tenancy, and branding standards for the PaVa-EDUX platform. All development, database migrations, and UI components must strictly adhere to these rules.

---

## 🏛️ 1. The "Anchor" Rule: School-Branch Prefix

Every identifier generated by the system—whether for a student, staff member, or financial record—must lead with the `SCHOOL-BRANCH` protocol. Technical database IDs (UUIDs or serials) must never be visible to the user as primary identifiers.

### 🧩 Standard Identifier Format
`[SCHOOLCODE]-[BRANCHCODE]-[TYPE]-[OPTIONAL_YEAR]-[SEQ]`

#### **Examples (Durga School | Sangareddy Branch)**:
- **School Code**: `DURGA`
- **Branch Code**: `SRD`

| Record Type | **Storage ID (Database)** | **Display ID (UI)** | Logic Breakdown |
| :--- | :--- | :--- | :--- |
| **Student (Permanent)** | `DURGA-SRD-STU-0000001` | `SRD-STU-0000001` | School-Branch-Type-Seq(7) |
| **Admission Number** | `DURGA-SRD-2026-27-STU-00001` | `SRD-STU-00001` | School-Branch-Year-Type-Seq(5) |
| **Fee Receipt** | `DURGA-SRD-2026-27-REC-00001` | `SRD-REC-00001` | School-Branch-Year-Type-Seq(5) |
| **Teacher ID** | `DURGA-SRD-TEAC-0001` | `SRD-TEAC-0001` | School-Branch-Role-Seq(4) |

## 🔐 2. Ultimate Tenancy Strategy: The Three-Layer Shield

To ensure **ZERO possibility** of cross-tenant data leakage, the system enforces isolation at three distinct levels of the stack.

### 🥉 Level 1: Application Filtering (Development Base)
Every query must lead with the mandatory tenant filters in the `where` clause.
- **Rule**: `where: { schoolId, branchId }` must be the first criteria.

### 🥈 Level 2: Middleware Enforcement (System Guard)
All Prisma queries are automatically intercepted by a global client extension that injects the current session's `schoolId` and `branchId`. 
- **Benefit**: A developer literally *cannot* forget the filter. It is an automated system guard.

### 🥇 Level 3: Database Isolation (PostgreSQL RLS)
The database itself enforces row-level security (RLS). 
- **Protocol**: The DB refuses to return any row if the `app.school_id` and `app.branch_id` from the session session don't match the row's ownership.
- **Goal**: 100% security even if the application code has a bug.

## 🚫 3. Critical Tenancy Guardrails

| Standard | Rule Detail | **Consequence** |
| :--- | :--- | :--- |
| **Immutable Fields** | Once created, `schoolId` and `branchId` can NEVER change for a record. | **Forbidden Move** |
| **API Guard** | No API shall accept a `schoolId` from the frontend; it must resolve from the server-side JWT. | **Security Lockdown** |
| **Context Lock** | Before any Insert/Update, the system must assert that `input.tenantId === context.tenantId`. | **Validation Block** |
| **Anti-Mixing** | Direct database access without a tenant context is forbidden. | **Audit Violation** |

## 👑 5. Access Hierarchy

The system distinguishes between three primary oversight levels:

1. **DEVELOPER (DEVP)**: Universal, cross-infrastructure access for global maintenance. (Strictly for Antigravity & Core Team).
2. **OWNER (OWNR)**: Management visibility across all branches and schools within their specific ownership group.
3. **STAFF (STAF)**: Branch-locked access. Operations are strictly restricted to the user's assigned branch.

Headers and summaries must always use human-readable branding.

- **❌ WRONG (Technical)**: `DURGA-SRD-...`
- **✅ CORRECT (Human)**: `Durga School | Sangareddy`

## ⚡ 7. Validation Standard: "The Atomic Guard"

To ensure data integrity and a premium user experience, all input surfaces must implement real-time validation.

- **Rule**: Every form field must have associated Zod schemas and UI-level feedback.
- **Timing**: Validations should be "Quick" (Instant).
- **Triggers**: Use `onChange` (typing) for formatting and `onBlur` (leaving field) for structural validation to prevent "Submission Shocks."
